вҸә The next phase is Milestone 5: evaluation and evidence validation вҖ” the last
  big pipeline milestone before we start exporting to the app. In plain English,
  this is where we stop just collecting promises and start checking them
  against reality.

  Here's what it involves, broken into the usual phases:

  Phase A вҖ” Human-review CLI (first, on purpose). A small terminal tool that
  shows you each stored promise with its quote and context, and lets you mark
  it: real promise / opinion / fragment / wrong topic. We run it over the 122 v2
  promises to get an actual precision number. This tells us whether the
  "opinion statements sneaking in" problem is 2% or 20% before we spend any GPU
  money on evaluation.

  Phase B вҖ” Evaluation stage. For each scoreable promise of an incumbent, the
  LLM gets the verified promise plus a pre-aggregated donor summary and a
  filtered vote list (each item carrying its database id), and returns a
  judgment like "kept / broken / mixed / no evidence" with the specific vote and
  donation ids it relied on.

  Phase C вҖ” Evidence validation (the trust gate). Code checks every cited id:
  the record must exist AND actually support the stated direction. Any
  evaluation with an invalid citation never ships вҖ” same philosophy as the quote
  gate in M4.

  Phase D вҖ” GPU run + inspection. Spin up the A100 with the proven gpu_serve.sh
  recipe, evaluate Collins and Pingree (our two incumbents with vote records),
  and review the results together.

  One note: only Collins and Pingree have voting records (Russell, Dunlap, and
  LePage are challengers), and Collins only has 2 stored promises from the pilot
  documents, so the evaluation sample will be small вҖ” that's fine for proving
  the pipeline before scaling.

  The Manifold update doesn't change anything until Phase D, but good to know
  it's ready.

вҸә Phase A is done: 53 tests passing (2 new), ruff and pyright clean, committed
  as 1aeb6ca and pushed on the new feature/milestone-5-evaluation branch.

  What got built, in plain English:

  - A verdict table (promise_reviews, migrationВ 0009). One row per promise,
  storing your judgment. Re-reviewing a promise overwrites the old verdict
  rather than piling up duplicates. Each verdict is tagged with the prompt
  version and model that produced the promise, so we can compare v2's precision
  against any future v3 directly.
  - The review tool. Run uv run python -m pipeline.review_cli and it walks you
  through all 122 promises, one at a time. For each one you see: who said it,
  the document title and link, the extracted quote highlighted between >>> <<<
  markers, and about 300 characters of surrounding transcript on each side so
  you can judge without opening YouTube. Then one keypress:
    - y correct (real promise, labeled right)
    - o opinion (a stance, not a commitment вҖ” the leak class we suspect)
    - f fragment (too thin to mean anything)
    - n not a promise (some other miss)
    - s real promise but wrong specificity tier, t real promise but wrong topic
    - k skip for now, q quit

  Every verdict saves immediately, so you can quit mid-session and pick up where
  you left off вҖ” the queue only shows unreviewed promises.
  - The precision report. uv run python -m pipeline.review_cli --report totals
  the verdicts and computes precision, counting s and t as real promises (the
  promise is genuine, only the label is off).

  One judgment call to note: I made the verdicts single-choice rather than
  letting you flag both "wrong topic AND wrong specificity" on one promise. With
  122 items, speed of review beats completeness of labeling вҖ” use the note
  field if a promise has two problems.

  Your move: run the CLI over the 122 promises (probably 20-30 minutes), then
  run --report and share the numbers. That precision figure decides whether we
  need a prompt v3 before the evaluation stage, or can proceed straight to Phase
  B.
